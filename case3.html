두 가지 운영 기준 정비를 병행했습니다. 답변 저장과 처리완료 상태가 구분되지 않아 후속 확인이 누락되던 CRM 구조를 점검·재정비하고, 환불·교환 의도를 필수 선택값으로 구조화해 고객 재문의 흐름을 제거했습니다.

Problem

무엇이 문제였는가

문제 1. CRM 상태값 미구분 — 후속 확인 누락
CRM에서 답변 저장 상태와 처리완료 상태가 구분되지 않았습니다. 실제로는 답변을 저장만 하고 발송하지 않은 건이 처리완료로 표시되거나, 후속 확인이 필요한 건이 완료 처리된 것으로 잘못 분류되어 누락되는 사례가 있었습니다.
문제 2. 환불·교환 의도 미수집 — 처리 전 재문의 반복
환불과 교환 요청이 접수될 때 고객 의도(환불인지 교환인지)가 수집 단계에서 확인되지 않아, CS 담당자가 처리 전 고객에게 다시 확인하는 왕복이 반복됐습니다. 리드타임이 늘어나고 고객 불편도 누적됐습니다.

What I Did

어떻게 접근했는가

CRM 정비
상태값 구조 점검 및 재정비 — 운영 기준 명확화
CRM 상태값 구조를 점검해 답변 저장 / 발송 대기 / 처리완료 단계가 명확히 구분되도록 정비했습니다. 후속 확인이 필요한 케이스가 완료 처리되지 않도록 운영 기준을 재수립하고, 담당자 교육을 병행했습니다. 이를 통해 후속 확인 건 누락 구조를 해소하고 운영 정확도를 높였습니다.
입력 구조화
환불·교환 의도를 필수 선택값으로 구조화 — 재문의 흐름 제거
환불·교환 요청 접수 단계에서 의도를 필수 선택값으로 수집하는 구조를 만들었습니다. 기존 입력 필드를 재활용하는 방식으로 개발 범위를 최소화해 빠르게 반영했습니다. 고객 의도가 접수 시점에 이미 확인되므로 처리 전 재문의 흐름이 제거되고, AS 처리 리드타임이 단축됐습니다.

Relevance

커머스 플랫폼 운영에 연결되는 지점

커머스 플랫폼에서는 상태값과 입력값이 곧 후속 처리 기준이 됩니다. 상태가 실제 처리 흐름을 반영하지 않거나, 고객 의도가 구조화되지 않으면 운영자는 같은 내용을 다시 확인하고 고객사는 처리 지연을 경험하게 됩니다. 이 케이스는 상태값과 입력 구조를 정리해 후속 누락과 재문의 흐름을 줄인 사례입니다.

Insights

이 케이스에서 배운 것

이 케이스에서 중요한 점은 CRM을 단순 기록 도구가 아니라 운영 흐름을 결정하는 기준으로 보았다는 점입니다. 고객 문의를 빠르게 닫는 것보다, 다음 담당자가 같은 상태를 같은 의미로 이해할 수 있게 만드는 것이 운영 정확도를 높인다고 생각합니다.
